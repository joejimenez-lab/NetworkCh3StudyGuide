§ 3.1

What the Transport Layer Does

The transport layer provides logical communication between application processes running on different hosts. Think of it this way:

Analogy (the book's classic): The postal service moves letters house-to-house. Ann and Bill (inside each house) move letters person-to-person. The postal service = network layer (IP). Ann & Bill = transport layer (TCP/UDP).

ConceptIn the analogyIn the network
MessagesLettersApplication data
ProcessesCousinsApplication processes
HostsHousesEnd systems
Transport protocolAnn & BillTCP / UDP
Network protocolPostal serviceIP

Key Facts

  • Transport protocols run only in end systems — routers don't touch transport headers.
  • IP is unreliable / best-effort: no guarantee of delivery, order, or integrity.
  • TCP builds reliability on top of unreliable IP. UDP does not.
Two Internet Transport Protocols

UDP (User Datagram Protocol) — connectionless, unreliable. Gives you only: multiplexing/demultiplexing + error checking. That's it.

TCP (Transmission Control Protocol) — connection-oriented, reliable. Gives you: mux/demux, error checking, reliable data transfer, flow control, congestion control, connection management.

§ 3.2

Multiplexing & Demultiplexing

Multiplexing (sender): Gather data from multiple sockets, wrap with transport headers, pass to network layer.
Demultiplexing (receiver): Deliver incoming segments to the correct socket.

How it Works — Sockets & Port Numbers

  • Each socket gets a port number (16-bit: 0–65535). Well-known ports: 0–1023 (HTTP=80, HTTPS=443, DNS=53, FTP=21).
  • A segment carries: source port, destination port (+ source/dest IP addresses at network layer).
🔵 UDP Demux

Uses only destination IP + destination port. Two different source IPs sending to the same dest port → same socket.

🟢 TCP Demux

Uses all four values: src IP, src port, dst IP, dst port. Same dest port but different src → different sockets. Web servers exploit this for per-client connections.

Common exam trap: A web server receiving two connections from different clients on port 80 uses different sockets for each, even though both have destination port 80. This is because TCP uses all four tuple values.

§ 3.3

Connectionless Transport: UDP

Why Use UDP?

  • No connection setup — no delay before sending.
  • No connection state — servers can support more clients.
  • No congestion control — can send at any rate (streaming media, DNS prefer this).
  • Small header overhead: 8 bytes vs TCP's 20 bytes.
  • Used by: DNS, SNMP, streaming video, online gaming, VoIP.

UDP Segment Structure

Source Port16 bits — port of sending process Dest Port16 bits — port of receiving process Length16 bits — length of entire UDP segment (header + data), in bytes Checksum16 bits — error detection

Total header = 8 bytes (4 fields × 2 bytes each).

UDP Checksum — How It Works

Uses 1's complement arithmetic:

1
Sender: Treats segment contents as 16-bit integers. Sums them all. Takes 1's complement of the result. Puts it in the checksum field.
2
Receiver: Sums all 16-bit words including checksum. If no errors → result is all 1s (i.e., 1111...1). If any bit = 0, there's an error.

The checksum can detect most errors but not all. Two errors can cancel out. The checksum can't correct errors — UDP just discards corrupt segments.

§ 3.4

Principles of Reliable Data Transfer (RDT)

These are the theoretical building blocks. Each version adds one more mechanism to handle a harder channel model.

The RDT Protocol Evolution

rdt 1.0
Perfect channel — nothing can go wrong

Channel is perfectly reliable. Sender just sends. Receiver just receives. No error handling needed. Unrealistic but foundational.

rdt 2.0 — ADDS: checksums + ACK/NAK
Channel with bit errors (no packet loss)

Detects corrupted packets via checksum. Receiver sends ACK (good) or NAK (bad). Sender retransmits on NAK. Stop-and-wait: sender can't send new data until ACK received. Fatal flaw: ACK/NAK can itself be corrupted!

rdt 2.1 — ADDS: sequence numbers (0 and 1)
Handles corrupted ACK/NAK

Sender adds 1-bit sequence number (0 or 1) to each packet. If ACK/NAK is garbled, sender retransmits. Receiver uses sequence number to detect duplicates. Also called the alternating-bit protocol.

rdt 2.2 — ADDS: NAK-free (ACK-only)
Same capability, no NAKs

Eliminates NAK. Instead, receiver sends ACK with the sequence number of the last correctly received packet. Duplicate ACK = implicit NAK for the next packet. (This is basically what TCP does!)

rdt 3.0 — ADDS: timer / timeout (handles packet loss)
Channel with bit errors AND packet loss

Sender adds a countdown timer. If no ACK within timeout, retransmit. Handles: lost data packets, lost ACKs, premature timeouts (→ duplicate packets, but sequence numbers handle those). This is the alternating-bit protocol, fully functional.

The Four Mechanisms (Memorize These)

Checksum

Detect bit errors in a packet.

Sequence Numbers

Detect duplicates, identify in-order delivery.

ACK / Timer

Confirm receipt; retransmit if timeout expires before ACK.

Retransmission

Re-send on timeout or NAK (or duplicate ACK).

rdt 3.0 Performance Problem

rdt 3.0 is stop-and-wait: terrible utilization. Example: 1 Gbps link, RTT = 30ms, packet = 1000 bytes:

Usender = (L/R) / (RTT + L/R) = 0.008ms / 30.008ms ≈ 0.00027 (0.027%)

That's only 267 kbps effective throughput on a 1 Gbps link! The fix is pipelining.

§ 3.4.2–3.4.4

Pipelining: GBN & Selective Repeat

Pipelining: allow the sender to transmit multiple unacknowledged packets simultaneously. Window size N = number of unACK'd packets allowed in flight.

🔄 Go-Back-N (GBN)
  • Sender window: N unACK'd packets
  • Receiver window: 1 (in-order only)
  • On timeout of packet n: retransmit n, n+1, ..., n+N-1 (all)
  • Receiver discards out-of-order packets (sends ACK for last in-order)
  • Uses cumulative ACKs
  • Single timer for oldest unACK'd packet
  • Sequence number space: 2k for k-bit field; window ≤ 2k–1
🎯 Selective Repeat (SR)
  • Sender window: N unACK'd packets
  • Receiver window: N
  • On timeout: retransmit only that packet
  • Receiver buffers out-of-order packets; delivers in-order to app
  • Uses individual ACKs per packet
  • Individual timer per unACK'd packet
  • Window ≤ 2k–1 (half the sequence space!) to avoid ambiguity

SR window size constraint: Window must be ≤ half the sequence number space. With sequence numbers 0–3 (4 values), window max = 2. Otherwise the receiver can't tell if an arriving packet is a retransmit of an old one or a new one.

Key Difference at a Glance

PropertyGBNSR

TCP is a hybrid of GBN and SR: cumulative ACKs like GBN, but buffers out-of-order segments like SR.

§ 3.5

TCP: Connection-Oriented Transport

Reliable Connection-Oriented Full-Duplex Point-to-Point Flow Control Congestion Control

TCP Segment Structure (Header = 20 bytes typical)

Source Port16 bits Dest Port16 bits Sequence #32 bits — byte-stream position of first byte in this segment ACK #32 bits — next byte the sender expects from the other side Header Length4 bits — length in 32-bit words Flags6 bits: ACK, RST, SYN, FIN, CWR, ECE, URG, PSH Receive Window16 bits — for flow control (rwnd) Checksum16 bits — error detection Urgent Pointer16 bits — rarely used in practice OptionsVariable — MSS negotiation, window scaling, timestamps

Sequence Numbers & ACK Numbers

  • Sequence numbers are byte-stream offsets, not packet numbers.
  • If a segment starts at byte 0 and has 1000 bytes of data → seq# = 0. Next segment → seq# = 1000.
  • ACK number = sequence number of the next byte expected. E.g., if bytes 0–535 received, ACK = 536.
  • TCP uses cumulative acknowledgments — ACK only up to first gap.
  • Initial sequence numbers are chosen randomly to prevent old segments from being misinterpreted.

Telnet Example: Client (ISN=42) sends 'C'. Segment: seq=42, ACK=79, data='C'. Server (ISN=79) echoes back: seq=79, ACK=43, data='C'. Client ACKs: seq=43, ACK=80, no data.

RTT Estimation & Timeout

EstimatedRTT = (1 – α) · EstimatedRTT + α · SampleRTT   [α = 0.125]
DevRTT = (1 – β) · DevRTT + β · |SampleRTT – EstimatedRTT|   [β = 0.25]
TimeoutInterval = EstimatedRTT + 4 · DevRTT
  • EWMA (Exponential Weighted Moving Average) — recent samples get more weight.
  • SampleRTT is never measured for retransmitted segments (Karn's algorithm).
  • On each timeout: double the TimeoutInterval (limited congestion control).
  • Initial TimeoutInterval = 1 second.

TCP Reliable Data Transfer

// Three key events at TCP sender: Data from app: create segment → start timer if not running → send to IP Timeout: retransmit oldest unACK'd segment → restart timer ACK received (y): if y > SendBase → update SendBase → restart timer if more unACK'd

Fast Retransmit

Don't wait for timeout — retransmit immediately on 3 duplicate ACKs.

  • Duplicate ACK = ACK for something already acknowledged (indicates gap in received data).
  • 3 dup ACKs = implicit signal that the segment after the triply-ACK'd one is lost.
  • Much faster than waiting for timeout (which could be ~1 second).
§ 3.5.5

Flow Control

Flow control prevents the sender from overflowing the receiver's buffer. Different from congestion control (which deals with the network).

rwnd = RcvBuffer – [LastByteRcvd – LastByteRead]
Constraint: LastByteSent – LastByteAcked ≤ rwnd
  • Receiver advertises rwnd (receive window) in every TCP segment it sends.
  • Sender keeps unacknowledged data ≤ rwnd.
  • Edge case: if rwnd = 0, sender keeps sending 1-byte segments to probe. Otherwise sender would be blocked forever when buffer empties.

Flow Control vs. Congestion Control: Flow control = receiver's buffer limit. Congestion control = network's capacity limit. Both throttle the sender but for different reasons. min(rwnd, cwnd) is the effective window.

§ 3.5.6

TCP Connection Management

Three-Way Handshake (Connection Setup)

1
Client → Server: SYN
SYN=1, seq=client_ISN. Randomly chosen initial sequence number (client_ISN). No data.
2
Server → Client: SYNACK
SYN=1, ACK=1, seq=server_ISN, ack=client_ISN+1. Server allocates buffers. No data.
3
Client → Server: ACK
SYN=0, ACK=1, seq=client_ISN+1, ack=server_ISN+1. Client allocates buffers. Can carry data.

Connection Teardown (4-Way FIN)

1
Client: FIN → enters FIN_WAIT_1
2
Server: ACK → client enters FIN_WAIT_2, server enters CLOSE_WAIT
3
Server: FIN → server enters LAST_ACK
4
Client: ACK → client enters TIME_WAIT (30s–2min), then CLOSED. Server moves to CLOSED.

TIME_WAIT: Client waits before fully closing to allow the final ACK to be resent if the server didn't receive it. This prevents old stale segments from confusing new connections.

SYN Flood Attack & SYN Cookies

Attacker sends many SYN segments without completing handshake → server allocates resources for half-open connections → server overwhelmed (DoS).

SYN Cookies (defense): Server doesn't allocate resources until the handshake completes. Instead, the server encodes connection state into the initial sequence number as a cryptographic hash (the "cookie"). Only allocates resources when a valid ACK arrives.

MSS — Maximum Segment Size

  • MSS = max application data per segment (not including TCP header).
  • Determined by MTU (maximum transmission unit) of local link.
  • Ethernet/PPP MTU = 1500 bytes → MSS typically 1460 bytes (1500 – 40 bytes for TCP/IP headers).
§ 3.6

Principles of Congestion Control

Costs of Congestion

  • Large queuing delays as sending rate approaches link capacity.
  • Wasted retransmissions — buffer overflow → packet drops → retransmit → wastes bandwidth.
  • Wasted upstream capacity — a packet dropped at the 2nd hop means the 1st hop's work was wasted.

Two Approaches to Congestion Control

End-to-End (TCP's approach)

Network gives no explicit feedback. End systems infer congestion from packet loss (timeout, triple dup ACK) and increasing delay.

Network-Assisted (ECN)

Routers explicitly signal congestion by marking bits in IP datagrams (ECN bits). Receiver relays signal to sender via ACK.

§ 3.7

TCP Congestion Control

TCP controls sending rate using the congestion window (cwnd):

LastByteSent – LastByteAcked ≤ min(cwnd, rwnd)

Send rate ≈ cwnd / RTT bytes/sec. Adjust cwnd to control rate.

The Three Algorithms: Slow Start, Congestion Avoidance, Fast Recovery

Phase 1: Slow Start
▶ Start: cwnd = 1 MSS
▶ On each ACK: cwnd += 1 MSS → doubles every RTT (exponential growth)
▶ Exit when: cwnd reaches ssthresh → switch to Congestion Avoidance
▶ Exit when: triple dup ACK → Fast Recovery
▶ Exit when: timeout → cwnd = 1 MSS, ssthresh = cwnd/2, restart Slow Start
↓ cwnd = ssthresh (reached slow-start threshold)
Phase 2: Congestion Avoidance
▶ On each ACK: cwnd += MSS × (MSS/cwnd) ≈ +1 MSS per RTT (linear growth)
▶ Exit when: timeout → cwnd = 1 MSS, ssthresh = cwnd/2, go to Slow Start
▶ Exit when: triple dup ACK → ssthresh = cwnd/2, cwnd = ssthresh + 3·MSS, go to Fast Recovery
↓ triple dup ACK (partial loss signal)
Phase 3: Fast Recovery
▶ On each dup ACK: cwnd += 1 MSS
▶ Exit when: new ACK → cwnd = ssthresh, go to Congestion Avoidance
▶ Exit when: timeout → cwnd = 1 MSS, ssthresh = cwnd/2, go to Slow Start

AIMD — The Big Picture

Ignoring slow start, TCP uses Additive Increase, Multiplicative Decrease (AIMD):

  • Additive Increase: +1 MSS per RTT in Congestion Avoidance (probing for bandwidth).
  • Multiplicative Decrease: halve cwnd on triple dup ACK loss event.
  • This creates the characteristic sawtooth pattern in cwnd over time.
Average throughput ≈ (0.75 × W) / RTT    (where W = cwnd at loss)

TCP Tahoe vs. TCP Reno

EventTCP Tahoe (older)TCP Reno (standard)
Timeoutcwnd=1, ssthresh=cwnd/2, Slow StartSame as Tahoe
Triple dup ACKcwnd=1, ssthresh=cwnd/2, Slow Startcwnd=ssthresh+3, ssthresh=cwnd/2, Fast Recovery

Reno is smarter: triple dup ACK means some packets are still getting through, so it's less severe than a timeout. Reno responds more gracefully.

TCP CUBIC

  • Modern default (Linux). Used by ~50% of major web servers.
  • After loss, ramps back up to Wmax quickly (cubic function), then slows near Wmax.
  • Key insight: after a loss, the network likely hasn't changed much — get back to Wmax fast.

ECN — Explicit Congestion Notification

  • Congested router sets ECN bits in IP header before dropping packets.
  • Receiver echoes congestion signal to sender via ECE bit in TCP ACK.
  • Sender reduces cwnd (as if triple dup ACK) and sets CWR bit in next segment.
  • Advantage: signal congestion before packet loss occurs.

Fairness

With K TCP connections sharing a bottleneck link of rate R, AIMD converges to approximately R/K each — equal share.

Fairness caveats: (1) UDP ignores congestion control → can crowd out TCP. (2) Applications using parallel TCP connections get unfair share. (3) Shorter RTT connections grow their window faster → get higher throughput at shared bottlenecks.

§ 3.8

QUIC — Quick UDP Internet Connections

QUIC is an application-layer protocol built on top of UDP, designed for HTTP/3. Combines TCP's reliability + TLS's security in fewer round trips.

Traditional (HTTP/2 over TCP+TLS)
  • 1 RTT TCP handshake
  • 1–2 RTT TLS handshake
  • HOL blocking: one lost packet stalls all streams
QUIC (HTTP/3 over QUIC/UDP)
  • Combined connection + crypto handshake (fewer RTTs)
  • Multiple independent streams
  • A lost packet blocks only its stream, not others

Key QUIC Features

  • Connection-oriented & secure: Built-in encryption (always). Handshakes combined.
  • Streams: Multiple independent streams per connection. Each stream has its own reliable delivery.
  • No HOL blocking: Unlike HTTP/2 over TCP, a dropped packet only affects its own stream.
  • Congestion control: Based on TCP NewReno (RFC 6582).
  • Application-layer: Can be updated faster than kernel-space TCP/UDP.

Quick-Fire Exam Facts

  • TCP is full-duplex (simultaneous bidirectional data)
  • TCP is point-to-point only (no multicast)
  • TCP seq#s count bytes, not packets
  • ACK# = next expected byte (not last received)
  • TCP uses cumulative ACKs
  • Fast retransmit triggers on 3 duplicate ACKs (4 ACKs for same data total)
  • Timeout doubles TimeoutInterval each time (exponential backoff)
  • MSS = max application data per segment (NOT including TCP header)
  • MSS typical = 1460 bytes (Ethernet MTU 1500 – 40 TCP/IP header)
  • SYN flood defense = SYN cookies
  • UDP has 8-byte header. TCP has 20-byte minimum header
  • GBN window ≤ 2k–1; SR window ≤ 2k–1
  • Slow start is not slow — it's exponential! It's "slow" only compared to pre-TCP era
  • TCP Reno: 3 dup ACKs → cwnd = ssthresh + 3, ssthresh = cwnd/2
  • TCP Tahoe: always goes to cwnd=1 on any loss
  • QUIC = application-layer protocol over UDP, for HTTP/3
